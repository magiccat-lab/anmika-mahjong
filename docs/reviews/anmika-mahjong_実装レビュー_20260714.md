## 総評

設計はかなり堅い。majiang-core を三麻+アンミカ拡張でラップし、lockstep 同期＋サーバ権限検証（牌山・サイコロは server 側 crypto 生成）、和了後の多段フロー（搶槓→ロン→副露→冬→金北→サイコロ→フィーバー）は winPipeline.ts のステージマシンに整理済み。R1〜R24 のレビュー修正痕跡とテスト量から、既知の破綻パターンは大半潰れてる。以下は残ってるものだけ。

## P0 — ルール到達不能（最優先）

### 1. 四槓子が成立不能（ドラ上限5が「初期ドラ2枚」と不整合）
- `shan3.ts:70` 初期表ドラは2枚切り出し（`removeNonHuaAt(raw, 4, 2)`）
- `shan3.ts:208` `gangzimo()` は `_baopai.length === 5` で throw（4麻標準の「初期1+カン4=5」をそのまま流用）
- 結果: **半荘全体でカン（開槓ドラ追加）は3回まで**。1人4回のカンが物理的に不可能で、`game3.ts:2446-2448` の四槓子ダブル役満処理は到達不能の死にコード
- さらに `shan3.ts:141` `drawNewDora()`（秋・カン共用）は上限ガードなし。**秋の効果でドラが増えるとカン可能回数がさらに減り**、秋2回発動後は2回目のカンで throw
- throw は `declareKan` 側の try/rollback（`game3.ts:3223` コメント参照）で吸収されるが、`getKanCandidates`（`game3.ts:3216`）は嶺上残数しか見てないので **カン候補は表示されるのに宣言すると無言で失敗する** — 嶺上枯渇時に直した bug 9 と同じ UX 破綻が baopai 側に残ってる
- 修正案: 上限を `初期2 + カン4 = 6`（秋分は別枠カウント or 上限を「カン由来の追加4回まで」に変更）、`getKanCandidates` に同条件のプリフィルタ追加

## P1 — ルール乖離・チート耐性

### 2. 面前役祝儀が「合算」でなく「最大値のみ」
- `huleChip.ts:437-439`: 混一色5 / 清一色10 / 二盃口15 を `menzenChip = Math.max(...)` で選択
- 混一色+二盃口の複合で 20枚 のはずが 15枚 になる
- また本役満チップ（`huleChip.ts:381-387` の dama 分岐、ツモ[5,5,7,9]/ロン[10,10,12,14]）とは独立枠のため、文書の想定例「大車輪=清一10+二盃口15+役満10=35枚」がどの経路でも再現できない
- 修正案: `menzenChip += 5/10/15` の合算化。役満との併算可否は文書側の裁定と揃える

### 3. サーバ権限: サイコロ系3アクションが誰でも実行可能
- `authority.ts:416-421` `validatePostWinAction`: `rollSaiKoroDice / selectSaiKoroCombo / advanceSaiKoro` は無条件 return null。しかも `rollSaiKoroDice` は `roundEnded` 前でも通る（417行）
- client 側には owner gate がある（`store.ts` R5 P1 #2）が、改造 client なら**他人のサイコロチャンスの出目宣言を書き換える・勝手に振って消化する・advance で丸ごとスキップする**が可能
- 出目そのものは `ws_server.ts` が `crypto.randomInt` で server 生成するので出目偽造は不可。潰れてるのは「値」で、「進行操作」が開いてる
- 修正案: authority 側で「現在 chance の owner == actor」を検証（client の gate と同じ条件をサーバに複製）

### 4. リーチ後の強制カン未実装
- 文書ルール: リーチ後にカン可能牌をツモったら（待ちが変わらない場合）カンは強制
- 実装: `declareKan` は待ちが変わる暗槓を**拒否**するガードのみ。強制する仕組みは grep でも 0 件（任意でスルー可能）
- カンドラ・嶺上・シュバ絡みで期待値が変わるので、対人戦だと「カンしない選択」が有利選択になり得る
- 修正案: リーチ中ツモでカン候補が合法1択のとき自動宣言（or UI 上スキップ不可）

### 5. アガリ止めが返り東条件を素通しできる
- `game3.ts:2983-2988` `canAgariyame`: 親 + 最終場 + jushu==2 のみで、点数条件なし
- 返り東（`game3.ts:596-600, 3021-3025`: 全員40000未満なら東へ巻き戻し継続）はオーラス流局/和了の nextRound 経路で判定されるが、`agariyame()` は `state.finished=true` 直行
- つまり**全員40000未満でも親の和了→アガリ止めで半荘を終わらせられる**。返り東を強制ルールとするなら矛盾
- 修正案: `canAgariyame` に `トップ点 >= 40000` を追加

## P2 — 裁定確認・軽微（文書と実装の突き合わせが要るもの）

### 6. ダブロンの供託・順位処理が「宣言順先取り」
- リーチ供託は `game3.ts:2792` で applyHule 順の先頭者が総取り（`store.ts:728` の ronResults 順=宣言順）。上家取り/頭ハネのどちらでもない
- `authority.ts:42-45` `chooseWinnerByOya` は「親がいれば親、いなければ**最後**の宣言者」を lastWinner に採用
- 文書側にダブロン裁定の明文がなかったはず（総点検リスト該当項目）。裁定を決めて両方揃えるべき

### 7. サイコロチャンスの基礎値テーブル
- 実装: カラス140（`game3.ts:2389`）/ 八連荘140（2393）/ オールスター70（2414）/ 八華は副露35・面前70×2回（2628）/ 四華四北70×2（2633）/ 八華四北70×3（2642）/ 本役満アガリ70×役満数（2677）
- 天和・地和・人和は「本役満アガリ70×count」に畳まれていて個別140の区別なし。文書側で天和系を140にするならここが乖離
- 四華四北・八華四北が `shuvariApplicable=false` なのも文書の「シュバサイコロ扱い」一覧と要突き合わせ

### 8. ゾロ目連続特典の発動条件が「シュバ宣言者」限定
- `store.ts:1071-1074`: `game.shuvariActive[chanceWinner]` のみで判定。chance 側の `shuvariApplicable` フラグはゾロ目判定に使われず、モーダルの「シュバ非適用」表示にしか効いてない
- カラス等を「シュバサイコロ扱い」とする文書解釈なら、シュバ未宣言でもゾロ目特典が付くべきかの裁定が要る（05-12 リョー改定「シュバの役割はゾロ目連続判定の発動のみ」コメントとどちらが最新か）

### 9. サイコロ chip の mode=ron でぽっち倍率が消える
- `chip.ts:71`: mode='ron' かつ非フィーバーはぽっち倍率 bypass（05-21 リョー仕様）
- 一方 `store.ts:1108` コメントは「ぽっち倍率は**常に適用**」（05-12 仕様）のまま chance.mode を渡してる。ロン和了由来のサイコロチャンスだけぽっち倍率が乗らない。どっちの仕様が生きてるか確認して、コメントと挙動を揃えるべき

### 10. オープンリーチ周り
- 3人目のオープンをブロック（`openLizhi.size >= 2`）— 文書に記載のない制限
- オープンは供託2000点必要（`game3.ts:3481-3486`）— 「持ち点0でもリーチ可」裁定を採るなら乖離
- 押し出し本役満は「非リーチ者の放銃」全部に発動（`game3.ts:2254-2260`）— 文書の発動条件と一致か確認

### 11. シュバリの半荘1回が返り東で復活しない
- `game3.ts:3056` に「shuvariUsed は半荘累積、reset しない」と明示コメントあり。返り東=東1からやり直しでも未復活。意図なら文書に明記、意図でないなら返り東時 reset

### 12. 混老対の加算方式
- `game3.ts:2327-2331`: 混老頭・七対子の既存翻に**さらに** +6翻（対々込み+8翻）を push。文書の「6翻」が合計値なら過剰加算

### 13. トントンブーの親除外
- `game3.ts:667+`: 飛び判定が `p !== oya` 限定。通常は親アガリで親は飛ばないから無害だが、**逆ぽっち反転で親アガリでも親が失点するケース**では対象外になる。エッジだが裁定を

### 14. 開槓・秋の新ドラに華スキップなし
- 初期ドラは `removeNonHuaAt`（`shan3.ts:70`）で華を避けるのに、`drawNewDora`（141-148）は素通し。華牌がドラ表示牌に載る。初期と同じ扱いに揃えるか、華ドラ許容を文書に明記

### 15. gold.ts の二重 consume（現行実害なし・将来の地雷）
- `gold.ts:85-97` の分岐チェーンで consumeGold した後、`99-102` の `physicalSuit === suit` ブロックが**無条件にもう一度** consumeGold する構造
- ただし金牌はプール上 各スート1枚（gp/gs/gN、`shan3.ts:generateTilePool`）なのでカウンタ最大1、2回目は常に no-op で**現行構成ではバグとして観測されない**
- でかぽっち等の拡張で金が複数枚化した瞬間に発火する。99-102 のブロックを分岐チェーンに統合しておくべき
