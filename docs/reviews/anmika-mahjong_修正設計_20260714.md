## 2. アーキテクチャ方針

sol の6提案を**全面採用**する。異論なし:
1. 単一決定論 reducer（`commandId + matchId + roundId + expectedVersion`、同一IDは同一ACK）
2. Phase discriminated union（boolean 並立廃止、`legalActionsBySeat` 経由でのみ受理）
3. PhysicalTile ID 116枚（zone 一意、core 変換は境界の一時 view）
4. `evaluateWin`（副作用なし）/`settleClaims`（供託・本場・倍率・逆ぽっち・トビ・祝儀を一括確定）分離
5. 単一権威サーバー+SQLite 永続（FastAPI は認証・ロビーのみ）
6. UI・アニメを進行条件から分離（サイコロ結果はサーバー先行確定、3Dは表示のみ）

俺からの補強2点:
- `legalActionsBySeat` に**サイコロ現chance owner 検証**を明示的に含める（authority.ts:416 の穴を reducer 仕様に昇格）
- **カンの合法判定にドラ上限を含める**（候補生成と宣言受理が同じ述語を使う。「候補は出るが宣言で失敗」類を仕様レベルで禁止）

## 6. 実装順序

```
WS-A（止血9件、1パス）
  → WS-B1 PhysicalTile（B の土台）
  → WS-B2-B4 evaluateWin/settleClaims
  → WS-C（オンライン一本化。B と並走可、ただし C2 の snapshot schema は B1 確定後）
  → WS-D 牌譜
  → release gate 一式で判定
```

- 順序根拠: A は即効かつ全て局所。B1 が B2 以降と C2 永続化スキーマの前提。C は最大工数だが B と独立に着手可能（進行層はルール計算を呼ぶだけ）
- sol の言う通り big-bang 禁止。既存 UI・画像・ルール表示・使えるテストは残して core と online boundary を段階差し替え
- **WS-A 完了までは現行オンライン版を対人常用しない**（sol の運用判断に同意。ローカル試遊のみ、牌譜復元は使わない）
