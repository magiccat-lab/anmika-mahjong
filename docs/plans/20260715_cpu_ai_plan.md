# CPU対戦強化(押し引き+鳴き) 実装プラン (2026-07-15)

レビュー担当が起草。オーナー承認後に自走実装へ発注する。
**依存: テスト決定論化プラン(20260715_test_determinism_plan.md)の完了後に着手**(fixture テストがシード固定前提のため)。

## 目的・背景

- 現行 CPU は前進打牌(受け入れ×向聴)と安牌評価(現物>スジ>壁)は持つが、「自手の価値と放銃リスクの天秤」= 押し引き判断が無く、鳴き(ポン/明槓)は一切しない
- 対戦相手としての手応えを一段上げる。オーナー要望 2026-07-15「CPU対戦はやりたい」

## スコープ

- In: 押し引き判断、最小限の鳴き判断、CPU思考の速度改善、アンミカ固有の価値評価の軽い反映
- Out (non-goals / 触るな):
  - ルール・精算ロジックの変更(cpuActions/pickBestDiscard と store の CPU 分岐のみ触る)
  - 機械学習・外部AIライブラリ導入(ヒューリスティック拡張に留める)
  - 人間側 UI の変更
  - `game3.ts:679-686` の最終成績計算

## 要件(検証可能な形で)

1. **押し引き**: リーチ者が居る局面で、自手が(a)ノーテンかつ危険牌しか無い→安牌優先(ベタオリ)、(b)テンパイかつ待ち残数>=4 or 打点期待が高い(満貫級以上 or フィーバー中)→前進継続、の判断関数を持つ。判断は純関数(`decidePushFold(ctx) -> 'push'|'fold'`)としてfixtureテストで固定する
2. **鳴き**: awaitingFulou の CPU 解決で、以下のみ鳴く: (a)三元牌ポン(役確定) (b)鳴いて向聴が進み、かつ既に役が確定している形(断幺確定/トイトイ進行)。それ以外はスキップ。fixture テストで「鳴く局面/鳴かない局面」を各2つ以上固定
3. **アンミカ考慮**: 押し引きの打点期待にサイコロ役の基礎期待値(本役満70/カラス140等の係数)とフィーバー中倍率を含める(粗い係数で可、定数はテーブル化して調整可能に)
4. **速度**: pickBestDiscard + 押し引きを含む CPU 1手の思考が平均 50ms 以下(現行計測値を README or plan 追記でベースライン記録してから改善)
5. **回帰なし**: 既存 vitest 全 green (`--retry=0`、決定論化後)。CPU 3人で 20 局完走するスモークテストを追加し、エラー・スタックなしで完走する
6. 新ロジックは feature flag 不要(常時有効)だが、判断テーブルの定数は 1 箇所に集約し、コメントで意味を書く

## 制約

- コミット境界: 押し引き / 鳴き / 速度 / スモーク で分割(単独 revert 可)
- 押し引き・鳴きの「正解」が曖昧な局面はテストで固定せず、明確な局面のみ fixture 化する
- 判断に迷う仕様(例: 鳴きの許容範囲を広げるか)は実装せず相談に積む
